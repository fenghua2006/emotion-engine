# Roadmap: 如何越来越完善

## v0.1 ✅ 完成
多通道引擎核心——8通道独立、快慢双速、交互矩阵、认知评估

## v0.2 ✅ 完成
去除硬上限、弹性衰减、感知压缩、指数级冲击感

## v0.3 — 记忆-情绪耦合

```
当前情绪 ──→ 记忆存储优先级（闪光的更重）
当前情绪 ──→ 记忆召回偏向（悲伤更容易想起悲伤的事）
旧记忆   ──→ 持续微拉情绪基线（saga 效应）
```

**核心函数**:
- `store_memory(event, state)` → 计算情绪强度 → 决定存储层级
- `recall_memory(state)` → 当前状态相似度排序 → 返回最相关记忆
- `apply_memory_baseline(memories)` → 长期记忆对基线的累积偏移

**预计**: 80 行 Python，不依赖任何数据库

## v0.4 — 多角色交互

两个 EmotionEngine 实例之间的情绪感染:

```
A 的情绪输出 ──→ 作为 B 的 appraisal 输入
B 的情绪输出 ──→ 作为 A 的 appraisal 输入
```

研究基础: Self-Excited Dynamics (2024) — 同伴互动驱动情绪是被动观看的 3-5 倍

**预计**: 50 行胶水代码

## v0.5 — 记忆持久化 + 可视化

- SQLite / ChromaDB 存储记忆
- 情绪曲线 Web 面板（一个 HTML 页面）
- 角色配置 → 一键导出/导入 JSON

## v0.6 — MCP 协议封装

让任何 MCP 客户端能直接调情绪引擎:
```json
{
  "mcpServers": {
    "emotion": {
      "command": "uv",
      "args": ["run", "python", "-m", "emotion_mcp"]
    }
  }
}
```

## v0.7 — 角色市场

- 社区贡献的角色蒸馏配方
- 一个 JSON 文件 = 一个角色
- GitHub 上直接 PR 新角色

## v1.0 — 生产就绪

- 性能: <1ms per tick
- 稳定性: 永远不会 NaN/Inf/overflow
- 文档: 每个函数有中文注释
- 测试: 50+ 角色场景覆盖

## 三条原则

1. **不做过度设计**——每个版本只加一个核心功能，调稳了再往前走
2. **每版必须有测试角色**——心海是基准，新版本不能让她"不像自己"
3. **胶水编程**——MCP 用 ego-mcp 改，持久化用 ChromaDB，不自己造
